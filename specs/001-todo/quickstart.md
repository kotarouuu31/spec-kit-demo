# クイックスタートガイド: 個人用タスク管理ToDoアプリ

**機能**: 001-todo | **日付**: 2025-09-07

## 開発環境のセットアップ

### 前提条件
- Node.js 18+ 
- npmまたはyarn
- テスト用のモダンウェブブラウザ

### インストール
```bash
# リポジトリをクローンしてプロジェクトに移動
cd todo-app

# 依存関係をインストール
npm install

# 開発サーバーを起動
npm run dev

# ブラウザでhttp://localhost:5173を開く
```

## ユーザージャーニーテスト

### 1. 初回アプリ起動
**テスト手順**:
1. ブラウザでアプリを開く
2. 空状態メッセージが表示されることを確認
3. 「タスクを追加」ボタンが見えることを確認
4. コンソールにエラーが表示されないことを確認

**期待結果**: 明確なコールトゥアクションのあるクリーンな空インターフェース

### 2. 初回タスク作成
**テスト手順**:
1. 「タスクを追加」ボタンをクリック
2. テキスト入力、優先度選択、日付ピッカーのあるフォームが表示されることを確認
3. テキストフィールドに「食材を買う」を入力
4. 優先度を「高」に設定
5. 期限を今日に設定
6. 「保存」をクリック

**期待結果**: 
- タスクがメインリストに表示される
- タスクが高優先度として表示される（赤いインディケーター）
- 期限が「今日」として橙色で表示される
- 成功通知が表示される
- フォームが自動的に閉じる

### 3. 複数タスクの追加
**テスト手順**:
1. タスク「歯医者に電話」を追加 - 中優先度、期限なし
2. タスク「プロジェクトを完了」を追加 - 高優先度、明日期限
3. タスク「家の掃除」を追加 - 低優先度、来週期限

**期待結果**:
- タスクが優先度順に表示される: 高 → 中 → 低
- 同じ優先度内では新しい順
- 異なる優先度インディケーターが見える
- 期限ハイライトが正しく動作する

### 4. タスクの完了  
**テスト手順**:
1. 「食材を買う」のチェックボックスをチェック
2. 視覚的フィードバック（取り消し線、薄い表示）を確認
3. 完了タスクが完了セクションに移動することを確認
4. ヘッダーの完了数が更新されることを確認

**期待結果**:
- 完了時に即座の視覚的フィードバック
- タスクが移動するが動機付けのため表示は維持
- 統計が正しく更新される
- チェックを外してアクティブ状態に戻せる

### 5. 既存タスクの編集
**テスト手順**:
1. タスクテキスト「歯医者に電話」をクリック
2. フォームが現在の値で開くことを確認
3. テキストを「歯医者にクリーニングの電話」に変更
4. 優先度を高に変更
5. 明日の期限を追加
6. 変更を保存

**期待結果**:
- フォームが既存データで事前入力される
- 変更が正しく保存される
- タスクが新しい優先度に基づいて再ソートされる
- 更新タイムスタンプが変更を反映

### 6. タスクのフィルターとソート
**テスト手順**:
1. 優先度フィルターを使用して「高」優先度のみ表示
2. 高優先度タスクのみが表示されることを確認
3. ソートを「期限」に変更
4. タスクが日付順に再ソートされることを確認
5. 「完了済みを表示」をトグル
6. 完了タスクが表示/非表示されることを確認

**期待結果**:
- フィルターがページリロードなしで即座に動作
- ソートオプションでタスク順が正しく変更される
- 完了タスクの表示/非表示が適切に切り替わる
- フィルター状態がセッション中維持される

### 7. 期限切れタスクの処理
**テスト手順**:
1. システム日付を翌日に変更（またはブラウザ開発ツールを使用）
2. アプリを更新
3. 昨日のタスクが「期限切れ」として赤で表示されることを確認
4. 期限切れタスクを完了
5. 期限切れスタイルが消えることを確認

**期待結果**:
- 期限切れタスクが赤で明確にハイライト
- 期限切れ状態が動的に更新される
- 完了した期限切れタスクは期限切れスタイルを失う

### 8. データ永続化
**テスト手順**:
1. 異なる優先度/日付の複数タスクを追加
2. ブラウザタブを閉じる
3. アプリを再度開く
4. すべてのタスクが正しいデータで存在することを確認

**期待結果**:
- 全タスクデータがブラウザセッション間で永続化
- 優先度、期限、完了状況が維持される
- データロスなし

### 9. エラーハンドリング
**テスト手順**:
1. 空のテキストでタスクを保存 - バリデーションエラーが表示されること
2. 期限を過去の日付に設定 - 警告が表示されること
3. データベースエラーをシミュレート（切断） - エラーメッセージが表示されること
4. 非常に長いテキスト（500文字超）でタスクを作成 - 切り捨てまたは警告されること

**期待結果**:
- バリデーション問題に対する親切なエラーメッセージ
- データベース問題の適切な処理
- ユーザーが破綻状態に残されない
- 問題修正方法の明確な案内

### 10. モバイル/レスポンシブテスト
**テスト手順**:
1. モバイルデバイスまたはブラウザモバイルビューでアプリを開く
2. タッチターゲットが十分な大きさ（最小44px）であることを確認
3. 仮想キーボードでフォーム入力が機能することをテスト
4. 小画面で全機能がアクセス可能であることを確認
5. 横向き/縦向き方向変更をテスト

**期待結果**:
- アプリが異なる画面サイズに適応
- 全機能がモバイルでアクセス可能
- 良好なタッチ体験
- 横スクロール不要

## パフォーマンス検証

### 読み込み時間テスト
- 初期アプリ読み込み: < 2秒
- データベース操作: < 100ms応答時間
- UI更新: < 50ms変更描画
- 1000タスク: アプリが依然として応答性を保つ

### メモリ使用量テスト
- 新規アプリ読み込み: < 10MBメモリ使用量
- 100タスクあり: < 15MBメモリ使用量
- 1000タスクあり: < 50MBメモリ使用量
- 長時間使用後のメモリリークなし

### データベースパフォーマンス
- タスク作成: < 10ms
- 全タスク読み込み: 1000タスクで< 50ms
- タスク検索: 1000タスクで< 100ms
- 複雑なフィルター/ソート: 1000タスクで< 200ms

```bash
npm run build
npm run preview
```

**検証**:
- ビルドがエラーなしで完了
- アセットが適切にバンドル・最適化される

- 静的ファイルを任意のHTTPサーバーから提供可能
- 静的ホスティング以外のサーバー側要件なし
- データベースファイルが正しい場所に作成される
- 初期読み込み後はオフラインで動作
