# 技術調査: 個人用タスク管理ToDoアプリ

**機能**: 001-todo | **日付**: 2025-09-07

## 技術選択の決定

### ビルドツール: Vite
**決定**: Viteを開発サーバーとビルドツールとして使用
**理由**: 
- 開発時の高速HMR（ホットモジュールリプレースメント）
- vanilla JSプロジェクト向けの設定不要セットアップ
- CSS/JSの組み込みバンドルと最適化
- モダンESモジュールサポート
- シンプルな単一ページアプリにはwebpackより軽量

**検討した代替案**:
- webpack: シンプルな単一ページアプリには複雑すぎる
- Parcel: 良いがViteの方がvanilla JSサポートが優秀
- バンドラーなし: 手動でのアセット管理が必要になる

### ストレージ: better-sqlite3を使用したSQLite
**決定**: ローカルデータ永続化にbetter-sqlite3を使用
**理由**:
- 同期APIがシンプルなアプリ要件にマッチ
- サーバーセットアップ不要
- 個人使用規模（1000タスク未満）での高速読み書き
- 他のNode.js SQLiteライブラリより高いパフォーマンス
- 後でElectron/デスクトップ環境にも対応可能

**検討した代替案**:
- localStorage/IndexedDB: クエリ機能が限定的、リレーショナルデータ不可
- sqlite3パッケージ: 非同期APIで複雑性が増加
- 外部データベース: 個人使用アプリには過剰

### フロントエンド: vanilla JavaScript
**決定**: 最小限の依存関係でvanilla ES2022 JavaScriptを使用
**理由**:
- 最小限ライブラリ使用の要件にマッチ
- シンプルなCRUD操作にフレームワークオーバーヘッドなし
- 理解・保守が容易
- 高速な初期読み込み時間
- モダンJS機能（async/await、モジュール）で良好な開発体験

**検討した代替案**:
- React/Vue: シンプルなタスク管理には重すぎる
- Alpine.js: vanilla JSで十分な場合の余計な依存関係
- Lit: 良い選択肢だが依然として追加依存関係

### テスト: Vitest
**決定**: 単体・統合テストにVitestを使用
**理由**:
- Viteとのネイティブ統合
- 優れたESモジュールサポートを持つJest互換API
- 高速なテスト実行
- SQLiteデータベーステスト用の優秀なモック機能

**検討した代替案**:
- Jest: ViteでのESM設定に追加構成が必要
- Node.js組み込みテスト: 統合テスト要件には基本的すぎる

### 日付処理: date-fns（必要に応じて）
**決定**: 必要な場合のみdate-fnsを日付操作に使用
**理由**:
- ツリーシェイキング対応（必要な機能のみインポート）
- 日付変更バグを防ぐイミュータブルAPI
- 相対日付計算でネイティブDateより優秀
- 複雑性が低い場合はネイティブDate APIも使用可能

**検討した代替案**:
- ネイティブDate API: 基本的な日付操作には十分
- moment.js: 大きすぎかつレガシー
- dayjs: 軽量で良いオプションだが、date-fnsの方がツリーシェイキングが優秀

### CSSアーキテクチャ
**決定**: CSS GridとFlexboxを使用したモダンCSS
**理由**:
- シンプルな3画面レイアウトにCSSフレームワーク不要
- タスクリストレイアウトにはCSS Gridが最適
- テーマ用のCSSカスタムプロパティ
- CSS Grid/Flexboxでのレスポンシブデザイン内蔵

**検討した代替案**:
- Tailwind CSS: ビルド複雑性とバンドルサイズが増加
- Bootstrap: 最小限インターフェース要件には重すぎる
- CSS-in-JS: JavaScript複雑性が増加

## 実装パターン

### データベーススキーマ設計
- tasksテーブル単体、カラム: id, text, priority, due_date, completed, created_at
- 自動インクリメントIDにINTEGER PRIMARY KEYを使用
- シンプルさのため日付をISO文字列で保存
- 真偽値は0/1を使用（SQLite標準）

### 状態管理
- 複雑な状態管理は不要
- vanilla JSでの直接DOM操作
- 単一の真の情報源: SQLiteデータベース
- リアクティブ更新: データ変更時に影響するUIコンポーネントを再描画

### エラーハンドリング
- 全てのSQLite操作でtry/catchブロック
- 一般的なシナリオ向けのユーザーフレンドリーエラーメッセージ
- デバッグ用のコンソールログ
- 重要でない機能の優雅な劣化

### パフォーマンス考慮事項
- 遅延読み込み: 最初は表示可能なタスクのみ読み込み
- 検索・フィルタ操作のデバウンス
- バッチDOM更新にドキュメントフラグメントを使用
- WHERE句で使用されるSQLiteカラムにインデックス

## アーキテクチャ決定

### モジュール構造
```
src/
├── main.js           // アプリ初期化
├── db/
│   ├── database.js   // SQLite接続とスキーマ
│   └── queries.js    // SQLクエリと操作  
├── ui/
│   ├── taskList.js   // タスクリスト描画
│   ├── taskForm.js   // タスク追加・編集フォーム
│   └── filters.js    // 優先度・日付フィルタリング
├── utils/
│   └── dates.js      // 日付フォーマットユーティリティ
└── styles/
    ├── main.css      // グローバルスタイル
    └── components.css // コンポーネント固有スタイル
```

### コンポーネント間通信
- 直接関数呼び出し（シンプルアプリにイベントシステム不要）
- モジュール間での共有データベース接続
- UIモジュールはrender/update関数をエクスポート
- main.jsでの中央アプリ状態調整

この調査により、選択した技術スタックが要件に適していることを確認: 最小限の依存関係でシンプル、高速、オフライン対応の個人タスク管理アプリケーション。
